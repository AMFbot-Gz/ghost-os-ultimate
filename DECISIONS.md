# Ghost OS Ultimate — Architectural Decisions

Chronological log of significant architectural choices, with rationale.
Each decision records what was chosen, what was rejected, and why.

---

## ADR-001 — Node.js Queen + Python FastAPI layers (not a monolith)
**Date:** 2026-03 (initial design, PICO-RUCHE lineage)
**Status:** Active

**Decision:** The system is split into a Node.js gateway (port 3000) and 7 Python FastAPI microservices (ports 8001-8007). They communicate over localhost HTTP.

**Rationale:**
- Node.js excels at the MCP protocol, WebSocket fan-out, and the skills/tools ecosystem (npm). Keeping the API gateway in Node.js preserves compatibility with Claude Code MCP clients and browser-based dashboards without friction.
- Python is the natural language for ML, vision, embeddings, and system automation (ChromaDB, Ollama client, screenshot libs, subprocess wrappers). Forcing all of that into Node.js or all of Node.js into Python adds unnecessary translation layers.
- Each layer can be restarted independently without taking down the others. During development, you can hot-reload `brain.py` while `memory.py` keeps its ChromaDB connection alive.
- The split mirrors the cognitive model intentionally: Node.js is the nervous system relay; Python layers are the organs.

**Rejected:** A single FastAPI monolith. Rejected because it would couple restarts (a brain crash kills memory), create a single Python process owning all concerns, and lose the MCP/Node.js ecosystem.

**Rejected:** Full microservice deployment with Docker networking per layer. Rejected as premature for a single-machine agent; localhost HTTP is sufficient and avoids container orchestration overhead at this stage.

## ADR-002 — ChromaDB + nomic-embed-text (not sentence-transformers)
**Date:** 2026-03-16
**Status:** Active

**Decision:** Semantic memory uses ChromaDB `PersistentClient` with `nomic-embed-text` embeddings generated by the local Ollama instance (768-dimensional vectors). sentence-transformers was not used.

**Rationale:**
- `nomic-embed-text` runs inside Ollama, which is already a required dependency of the system (Ollama serves `llama3`, `llama3.2:3b`, `moondream`). Adding embedding generation costs zero new infrastructure — the same Ollama process handles it via `POST /api/embeddings`.
- sentence-transformers requires a Python model download (~400 MB), a separate process or in-process torch load, and adds significant cold-start latency. On the Intel i7-9750H target hardware, in-process torch embedding competes with FastAPI's async event loop.
- ChromaDB `PersistentClient` is file-backed (SQLite + parquet), zero-server, zero-config. It persists across restarts without a running database daemon.
- 768 dimensions from nomic-embed-text provides good semantic resolution for episode retrieval (mission text, outcomes, tool calls) without the memory cost of 1536D OpenAI-style embeddings.

**Rejected:** sentence-transformers + FAISS. Rejected due to in-process torch dependency and the availability of a simpler path through Ollama.

**Rejected:** OpenAI embeddings API. Rejected to keep the agent fully local and functional without internet access.

## ADR-003 — ReAct loop is additive, not a replacement for /think
**Date:** 2026-03-16
**Status:** Active

**Decision:** `POST /react` implements a new ReAct reasoning loop as a separate endpoint and code path. `POST /think` is preserved unchanged.

**Rationale:**
- `/think` is a single-shot structured decomposition: it takes a mission, returns a plan with subtasks, risk level, and an immediate action. This is appropriate for the Queen's vital loop, where the agent needs a structured response it can serialize, log, and act on deterministically.
- `/react` is a multi-step iterative loop (up to 15 steps) where the LLM drives its own observation-action cycle. It is appropriate for open-ended tasks where the correct sequence of actions is not known upfront.
- Replacing `/think` with `/react` would break the vital loop's structured output contract. The Queen parses specific JSON fields from `/think` responses; a free-form ReAct trace cannot substitute for that.
- Both endpoints coexist: the Queen dispatches to `/react` for exploratory missions and to `/think` for routine structured planning.

**Rejected:** Retrofitting ReAct into `/think` with a mode flag. Rejected because it would add conditional complexity to a well-tested endpoint and risk breaking the existing vital loop.

## ADR-004 — thinking=False for ReAct (anti-hallucination)
**Date:** 2026-03-16
**Status:** Active

**Decision:** `llm_react()` calls Claude with `thinking=False` (no adaptive thinking). The main `llm()` function uses `thinking=True` by default for planning calls.

**Rationale:**
- Claude's adaptive thinking mode produces extended reasoning tokens before the answer. For ReAct, the LLM output must follow a strict `Thought:/Action:/Action Input:/Observation:` format so that `_parse_react_step()` can extract structured fields with regex.
- When thinking is enabled, the model sometimes embeds its format decisions inside the thinking block and produces a compressed final answer, causing the parser to miss fields. This was the primary source of hallucinated or missing action fields during testing.
- Disabling thinking forces the model to produce the format in the visible output directly. The format becomes the reasoning scaffold — the Thought field serves the same exploratory purpose as the thinking block, but in a parseable location.
- Planning calls (`/think`) retain `thinking=True` because they benefit from extended deliberation before producing a structured JSON plan; the output is parsed with `json.loads()` which is tolerant of thinking preamble.

**Rejected:** Keeping thinking enabled and using a more permissive parser. Rejected because a permissive parser increases the surface area for the model to deviate from the format and act on hallucinated action inputs.

## ADR-005 — Critic is a separate LLM call (not part of the main loop prompt)
**Date:** 2026-03-16
**Status:** Active

**Decision:** After each non-read-only action in the ReAct loop, `critic_evaluate()` makes an independent LLM call using `CRITIC_SYSTEM_PROMPT`. The Critic is not a second agent in the same conversation thread.

**Rationale:**
- If the Critic's evaluation were injected into the main ReAct conversation as an Observation, the main agent would be evaluating its own action through the same context window — the same biases that caused the action would influence the evaluation.
- A separate LLM call with a dedicated system prompt creates genuine independence. The Critic sees only: the action, the action input, the execution result, and the expected outcome. It has no access to the main agent's reasoning chain, preventing rationalization.
- The Critic uses `llm_react()` (Claude, `thinking=False`) for the same format-consistency reason as the main loop — the response must parse to a clean JSON verdict (`rollback`, `retry`, `ok`) with a reason field.
- A separate call also means the Critic can be swapped for a different model or a lightweight rule-based evaluator without touching the main loop.

**Rejected:** Injecting a critic prompt as a system instruction into the ReAct conversation. Rejected because it conflates agent and evaluator context and makes the critic's verdict susceptible to the same biases it is supposed to detect.

## ADR-006 — Auto-rollback uses pattern matching first (before LLM)
**Date:** 2026-03-16
**Status:** Active

**Decision:** `_auto_rollback_cmd()` uses regex pattern matching to generate rollback commands for known destructive action patterns (file writes, moves, deletes, chmod, etc.) before any LLM is consulted. `_execute_rollback()` runs the result.

**Rationale:**
- For the most common destructive operations — writing a file, moving a file, deleting a file — the inverse is deterministic and does not require LLM reasoning. Asking an LLM to generate `rm newly_created_file.txt` when the action was `echo 'x' > newly_created_file.txt` adds latency and a point of failure for a trivial case.
- Pattern matching is synchronous, instant, and cannot hallucinate a rollback command. An LLM-generated rollback could produce a plausible-looking but incorrect command (wrong path, wrong flags) that causes secondary damage.
- The pattern matching covers the high-frequency cases (write, move, mkdir, chmod, install). Edge cases that do not match fall through: the Critic records the failure as unrollable and logs it, but does not attempt a speculative LLM rollback.
- This is a conservative safety posture: no rollback is safer than a wrong rollback.

**Rejected:** LLM-generated rollback commands for all cases. Rejected due to hallucination risk for high-stakes destructive operations.

## ADR-007 — JSONL is source of truth (not ChromaDB)
**Date:** 2026-03-16
**Status:** Active

**Decision:** `agent/memory/episodes.jsonl` is the canonical persistent store for all episodes. ChromaDB is a derived index, rebuilt from JSONL on demand via `POST /reindex` or on startup if the collection is empty.

**Rationale:**
- ChromaDB's `PersistentClient` is a SQLite + parquet stack. While robust for a local tool, it is not a write-ahead-log append store. JSONL is: each episode is one atomic `f.write(json.dumps(entry) + "\n")` call. Even a crash mid-write only corrupts the last line, which `_read_episodes_safe()` skips.
- JSONL is human-readable, greppable, and trivially diffable in git. ChromaDB's binary parquet files are not.
- If ChromaDB becomes corrupted or its schema changes between versions (as happened between ChromaDB 0.x and 1.x), the full index can be rebuilt from JSONL without data loss.
- The tradeoff is that `POST /reindex` is O(n) over all episodes. At the current scale (101 episodes, target 500 before archiving), this takes under 10 seconds on the i7-9750H. Revisit if episode volume grows to tens of thousands.

**Rejected:** Treating ChromaDB as the primary store and deriving JSONL from it. Rejected because ChromaDB's query API is not suitable for sequential reads, full exports, or the keyword-based fallback search that `POST /search` uses when ChromaDB is unavailable.
